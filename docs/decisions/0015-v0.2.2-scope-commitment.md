# 15. v0.2.2 scope commitment — release engineering polish + dependency hygiene

Date: 2026-04-29

## Status

Accepted

## Context

v0.2.1 shipped on 2026-04-28 with the following deferred items
explicitly tracked in the CHANGELOG.md `[0.2.1]` entry:

- **Performance**: PR 1 (postgres index migration cargo-cult,
  cancelled — indexes already existed); PR 3 (bulk IoC ingest
  endpoint + matcher background worker, YAGNI deferred — PR #47
  alone closed all v0.2.0 release-gate targets with margin).
- **Multi-ecosystem**: Rust sidecar parity for PyPI / Maven
  parsers (Wasm scope, v0.5.0+).
- **Dependency hygiene**: 5 GitHub Action major bumps still
  open (`cosign-installer` 3 → 4, `docker/login-action` 3 → 4,
  `actions/checkout` 4 → 6, `goreleaser/goreleaser-action`
  6 → 7, `toml` 0.x → 1.x in /native). All touch `release.yml`
  core surface — sequential PRs with snapshot validation
  required, not bundlable. Plus `kin-openapi` 0.135 → 0.137
  (zero functional value, deferred).
- **Known issues**: `TestPostgresContract` testcontainers
  shared-DB counter race produces intermittent
  `database "rampart_contract_<pid>_N" does not exist`
  failures on CI (observed PR #47 post-merge run + PR #59 pre-
  merge run; both passed on re-run). Backstage transitive
  security alerts (`uuid` < 14.0.0 GHSA-w5hq-g745-h8pq;
  `fast-xml-parser` < 5.7.0 GHSA-gh4j-gqv2-49f6) unresolved
  in `yarn.lock`.
- **Release engineering carry-over from v0.2.0**: image tags
  emitted as `:0.2.1` (semver-stripped) instead of `:v0.2.1`
  (semver convention); `engine` ghcr path lacks the `rampart-`
  prefix the other 4 published images carry; `Cargo.lock`
  fixture for cargo parser tests still uses legacy
  `simple.toml` filename, requiring `--ecosystem cargo` flag
  rather than auto-detect.
- **Documentation polish**: README "fifty times a day" phrasing
  flagged in v0.2.1 review as informal; one-line neutralization
  noted for next docs cycle.

The question is which of these belong in v0.2.2 vs v0.3.0+.

## Decision

v0.2.2 is a **release engineering polish + dependency hygiene**
release. **No new features. No new ecosystems. No performance
work.** Pure cleanup of items the v0.2.0 / v0.2.1 cycle
accumulated.

### In scope (v0.2.2)

**CI/test reliability**

- `TestPostgresContract` per-test database pattern. Replace
  the shared-container `freshDSN()` counter with one
  testcontainer per `t.Run` invocation, eliminating the
  `database "rampart_contract_<pid>_N" does not exist` race
  observed under parallel `go test` execution on CI runners
  (PR #47 post-merge run + PR #59 pre-merge run). Acceptance:
  50 consecutive
  `go test ./engine/internal/storage/postgres/... -count=10`
  runs green locally.

**Release pipeline normalization**

- Image tag normalization. `docker/metadata-action` config in
  `release.yml` currently emits `type=semver,pattern={{version}}`
  which strips the `v` prefix. Switch to `pattern={{raw}}` (or
  add a second tag) so consumers can pull `:v0.2.2` per semver
  convention. Backwards-compat: keep `:0.2.2` tag too for one
  release cycle; remove in v0.3.0.
- Engine ghcr path normalization. The `engine` image publishes
  to `ghcr.io/ugurcan-aytar/engine`, breaking the `rampart-*`
  prefix the other four images use. One-line matrix change in
  `release.yml`. Backwards-compat: dual-publish to both paths
  for one release cycle; deprecate the unprefixed path in
  v0.3.0.
- `Cargo.lock` fixture rename. Move
  `engine/testdata/lockfiles/cargo/{simple,medium}.toml` to
  `cargo/{simple,medium}/Cargo.lock` so the auto-detect
  filename rule fires, removing the `--ecosystem cargo` flag
  requirement that was flagged in v0.2.1 smoke testing.

**Dependency hygiene**

- 5 GitHub Action major bumps. Sequential PRs, each
  standalone, each gated on goreleaser snapshot dry-run +
  `act` local CI before merge. Non-bundlable so a single
  major can be reverted without unwinding the rest:
  - `sigstore/cosign-installer` 3.9.1 → 4.1.1 (revisit
    `SECURITY.md` cosign-verification recipe if v4 client
    commands diverge).
  - `docker/login-action` 3.7.0 → 4.1.0.
  - `actions/checkout` 4.3.1 → 6.0.2, skipping v5 (touches
    every workflow).
  - `goreleaser/goreleaser-action` 6.4.0 → 7.2.1 (highest-
    risk: the v0.2.1 CHANGELOG override step lives in the
    same job; snapshot dry-run on the PR branch is
    mandatory).
  - `toml` 0.8.23 → 1.1.2+spec-1.1.0 in `/native` (Rust
    crate, 0.x → 1.x API stability transition; cargo build
    + parity-test green required).
- Backstage transitive security alerts. Add `resolutions`
  block to `package.json` forcing `uuid: ^14`,
  `fast-xml-parser: ^5.7`. Validate via Backstage SPA mount +
  Playwright e2e (per the MUI 9 / Recharts breakage memory
  note: typecheck + lint + jest can pass while the SPA mount
  dies in production webpack). If the resolutions block
  surfaces unexpected breakage, scope expands — flag for
  re-discussion before merge.

**Documentation polish**

- README "fifty times a day" phrase neutralization.
  Single-line edit; no structural changes.

### Out of scope (deferred)

- **Bulk IoC ingest endpoint + matcher background worker**:
  YAGNI under current fleet sizes. Revisit in v0.3.0 alongside
  Theme H (multi-repo aggregation), where fleet size grows
  beyond the corpus PR #47 closed all targets against.
- **`kin-openapi` 0.135 → 0.137**: codegen-only transitive,
  zero functional value. Adds new transitive supply-chain dep
  (`santhosh-tekuri/jsonschema/v6`). Revisit when
  `oapi-codegen` requires the upgrade.
- **Multi-repo aggregation**: planned v0.3.0 Theme H, untouched.
- **Rust sidecar parity for PyPI / Maven**: Wasm-bridged
  parity ship target is v0.5.0+ (Theme M). Five-ecosystem
  CLI-only coverage acceptable until then.
- **New ecosystems** beyond the current five (npm, gomod, cargo,
  pypi, maven). The 5/8-dialect surface is the intended
  v0.2.x stopping point.
- **New `IoCKind` enums** beyond the current behavioural set
  (`packageVersion`, `packageRange`, `publisherAnomaly`).
  Per ADR-0011 amendment, ecosystem dimension flows through
  the `Ecosystem` field.

### Vocabulary note

v0.2.2 follows the v0.1.1 precedent for maintenance releases:
descriptive subsection headings rather than feature-release
"Theme X" letter labels. Theme letters span A through Y in
`ROADMAP.md` and are fully booked across feature releases
v0.2.0 through v0.9.0; introducing new letters mid-sequence
would force a cascading rename across many downstream themes.
The convention exists for PR shorthand on parallel
architectural workstreams (`Theme C1 + C2 — Go modules + Cargo
parsers`), which maintenance releases don't have. v0.1.1
already established the precedent — its `Blockers` /
`Nice-to-have` / `Deferred ecosystem upgrades` subsections
carry no Theme labels.

## Consequences

### Benefits

- v0.2.2 stays small and reviewable, ~1–2 weeks sustained
  tempo (vs v0.2.0's six themes / v0.2.1's two themes intense
  pace).
- Image tag + ghcr path normalization unblocks downstream
  consumers writing `:v0.2.2` per semver convention rather
  than the current semver-stripped `:0.2.2`.
- Backstage transitive alert closure removes the only open
  security advisory on the repository before any external
  adoption attempt.
- `TestPostgresContract` flake elimination raises CI
  reliability; eliminates the "re-run and hope" pattern
  observed twice in the v0.2.1 cycle.
- Five GitHub Action major bumps maintained without bundling,
  preserving the per-action revertability invariant.

### Costs

- No new user-facing features; release notes will read as a
  maintenance update. Discipline > rampage mode is a feature,
  not a bug.
- Backstage `resolutions` block (R2) carries known regression
  risk per the MUI 9 / Recharts breakage memory note —
  typecheck + lint + jest can all pass while production
  webpack mount dies. Playwright e2e validation gates the
  merge.
- Sequential PR discipline for five GH Action majors will
  consume 1–2 days of integration time. The non-bundle rule
  is non-negotiable: if a single major silently regresses
  the release pipeline, bundled PRs make the bisection harder.
