# Migrating from v0.1.x to v0.2.0

This guide covers operational changes required to upgrade an existing
rampart deployment from v0.1.x to v0.2.0. v0.2.0 is a major feature
release with three breaking changes that need explicit operator action.

## TL;DR â€” three required actions

1. **Provision Postgres** (or set `RAMPART_STORAGE=memory` to keep the
   v0.1.x in-memory backend).
2. **Configure JWT signing** if you enable auth â€” defaults remain off
   (`RAMPART_AUTH_ENABLED=false`) for backward-compat.
3. **Update Backstage plugin config** â€” `rampart.baseUrl` is replaced
   by backend proxy routing.

The rest of this document covers each in detail plus optional features.

## 1. Storage: in-memory â†’ Postgres (recommended)

v0.1.x ran on an in-memory store. v0.2.0 ships a Postgres backend
(production default) with a `memory` mode preserved for development.

### Postgres setup

`docker-compose.yml` ships a `postgres:16-alpine` service:

```yaml
postgres:
  image: postgres:16-alpine
  environment:
    POSTGRES_USER: rampart
    POSTGRES_PASSWORD: rampart
    POSTGRES_DB: rampart
  ports:
    - "5432:5432"
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U rampart"]
    interval: 5s
  volumes:
    - rampart-pg-data:/var/lib/postgresql/data
```

For self-managed Postgres, point the engine at it:

```bash
export RAMPART_STORAGE=postgres
export RAMPART_DB_DSN=postgres://rampart:rampart@postgres:5432/rampart?sslmode=disable
export RAMPART_DB_MAX_CONNS=10  # optional, default 10
```

Goose migrations run automatically on engine startup. Migration files
are embedded in the binary; no separate migration tool required. v0.2.0
ships seven migrations (components, sboms, iocs, incidents, publishers,
publisher_history, anomalies).

### Keeping in-memory mode

For development or testing, the in-memory backend is preserved:

```bash
export RAMPART_STORAGE=memory
```

In-memory state is lost on engine restart. Not recommended for
production.

### Data migration from v0.1.x

There is no automatic migration path. v0.1.x state was in-memory and
ephemeral by design. After upgrading, the engine starts with empty
storage; ingesting your fleet's SBOMs again is the expected first
operational step.

## 2. Auth posture (Theme A)

v0.2.0 ships JWT-based auth on `/v1/*` and an env-configured CORS
allow-list. Both default to off, preserving v0.1.x behaviour.

### JWT auth â€” defaults off

```bash
# Default: auth disabled (backward-compat with v0.1.x)
export RAMPART_AUTH_ENABLED=false
```

For production deployments, enable JWT:

```bash
export RAMPART_AUTH_ENABLED=true
export RAMPART_AUTH_SIGNING_KEY=<HS256-secret-or-RS256-public-key>
export RAMPART_AUTH_ALGORITHM=HS256  # or RS256
export RAMPART_AUTH_AUDIENCE=rampart-engine-prod  # this engine's identity
```

When auth is enabled:
- Mutation routes (`POST /v1/*`, `PUT`, `DELETE`, `PATCH`) require a
  valid Bearer token.
- Read-only routes (`GET /v1/*`) accept either authenticated or
  anonymous depending on per-deployment configuration.
- `POST /v1/auth/token` issues internal/test tokens (production
  deployments typically use an external IdP).
- Rejected tokens emit `AuthRejected` events on `/v1/stream` â€”
  signal for replay / brute-force detection.

See [`docs/operations/auth-providers.md`](../operations/auth-providers.md)
for IdP integration patterns (lands in v0.2.1).

### CORS â€” explicit allow-list

v0.1.x permitted all cross-origin requests. v0.2.0 defaults to
deny-all unless you configure origins:

```bash
# Default: deny all cross-origin (same-origin only)
unset RAMPART_CORS_ORIGINS

# Or: allow specific origins
export RAMPART_CORS_ORIGINS=https://backstage.example.com,https://dashboard.example.com

# Dev convenience (warns in logs):
export RAMPART_CORS_ALLOW_ALL=true
```

For Backstage-fronted deployments (recommended), no CORS config is
needed â€” frontend requests are same-origin against Backstage. The
Backstage backend plugin proxies to the engine.

## 3. Backstage plugin config (Theme B)

v0.1.x frontend client used `rampart.baseUrl` to call the engine
directly. v0.2.0 routes through the Backstage backend plugin.

### Frontend plugin config

Remove from `app-config.yaml`:

```yaml
# REMOVE this section
rampart:
  baseUrl: http://localhost:8080
```

The frontend now uses Backstage's `backend.baseUrl` + `/api/rampart/v1`
relative URLs.

### Backend plugin config

Add the engine connection details:

```yaml
rampart:
  engine:
    baseUrl: ${RAMPART_ENGINE_URL:-http://localhost:8080}
    authToken: ${RAMPART_ENGINE_AUTH_TOKEN}
```

`authToken` is required when `RAMPART_AUTH_ENABLED=true` on the
engine. The Backstage backend mints a service-to-service token at
startup (HS256 with the configured signing key) and sends it on every
proxied request. See
[ADR-0012](../decisions/0012-auth-boundary-at-engine.md) for the
auth-boundary-at-engine architecture.

### Backstage auth (deferred)

v0.2.0 still ships with `guest.dangerouslyAllowOutsideDevelopment`
enabled in the demo `app-config.yaml`. Theme A3 (replace guest auth
with real OAuth provider) is deferred â€” operators registering their
own GitHub OAuth App should follow the upcoming v0.2.1 OAuth setup
guide.

## 4. New optional features

### Publisher anomaly detection (Themes F1+F2+F3)

v0.2.0 introduces publisher metadata tracking and anomaly detection.
Both default disabled:

```bash
# Enable publisher metadata refresh (npm + GitHub APIs)
export RAMPART_PUBLISHER_ENABLED=true
export RAMPART_PUBLISHER_REFRESH_INTERVAL=1h
export RAMPART_PUBLISHER_BATCH_SIZE=50
export GITHUB_TOKEN=<optional-for-higher-rate-limits>

# Enable anomaly detection (requires publisher data)
export RAMPART_ANOMALY_ENABLED=true
export RAMPART_ANOMALY_DETECT_INTERVAL=1h
```

When enabled, the engine periodically refreshes publisher metadata
and detects three anomaly types: maintainer email drift, OIDC
publishing regression, version jump anomaly. Detected anomalies open
incidents through the standard workflow (see
[ADR-0014](../decisions/0014-anomaly-ioc-bridge.md) for the IoC
bridge design).

### New ecosystems (Theme C)

`rampart scan` now supports Go modules and Cargo in addition to npm:

```bash
rampart scan path/to/go.sum --ecosystem=gomod
rampart scan path/to/Cargo.lock --ecosystem=cargo
```

Auto-detection: filename `go.sum` â†’ gomod, `Cargo.lock` â†’ cargo,
`package-lock.json` â†’ npm.

PyPI and Maven parsers ship in v0.2.1.

## 5. Verification checklist

After upgrade, verify:

```bash
# 1. Postgres migrations applied
docker logs rampart-engine 2>&1 | grep "successfully migrated"
# Expected: "goose: successfully migrated database to version: 7"

# 2. Engine health
curl http://engine:8080/healthz
# Expected: 200 OK

# 3. (If auth enabled) token rejection works
curl -X POST http://engine:8080/v1/components
# Expected: 401 Unauthorized

# 4. (If auth enabled) valid token works
TOKEN=$(curl -X POST http://engine:8080/v1/auth/token \
  -d '{"subject":"verify","scope":"write"}' | jq -r '.access_token')
curl -X POST http://engine:8080/v1/components \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"ref":"test","ecosystem":"npm"}'
# Expected: 201 Created

# 5. Backstage frontend renders
# Browse to Backstage; rampart plugin tab loads incidents from
# /api/rampart/v1/incidents (relative URL, no CORS handshake)
```

## 6. Rollback plan

If v0.2.0 misbehaves, downgrade is supported but data is lost:

1. Stop v0.2.0 engine.
2. Start v0.1.x engine with the same `docker-compose.yml` (with
   Postgres service removed and `rampart.baseUrl` re-added to
   Backstage config).
3. Re-ingest SBOMs.

Postgres data persists across the downgrade but v0.1.x has no
storage backend that reads it â€” the data is preserved for re-upgrade
later.
